# Internationalization (i18n)

The CLI supports German and English output, selected via the `LANGUAGE`
environment variable at process start:

```bash
LANGUAGE=de pbd-toolkit scan .    # German output
LANGUAGE=en pbd-toolkit scan .    # English output
pbd-toolkit scan .                # unset / unsupported value -> English
```

Only `LANGUAGE=de` selects the German catalog. Anything else — unset, empty,
or an unsupported locale such as `fr` — falls back to English.

## Writing translatable strings

Use `core.i18n.get_translator()` to obtain a translation function at the
start of a Typer command, then route every user-facing string through it:

```python
from core import i18n

@app.command()
def my_command() -> None:
    translate_func = i18n.get_translator()
    typer.echo(translate_func("Something went wrong: {}").format(exc), err=True)
```

Use `str.format()` placeholders (`{}`, `{:.2f}`, ...) rather than f-strings for
any text that carries a translatable template — an f-string bakes the
interpolated value into the string *before* translation ever sees it, so the
literal template can't be looked up in the catalog.

The scan pipeline (`core/scan_reporting.py` and everything reachable through
`core.context.Context`) uses the same function via `context._(...)` /
`context.translate(...)`, threaded through as a constructor argument rather
than a global — this keeps scans testable without patching global state.

## Regenerating the catalog

The catalog lives in `locales/base.pot` (template) and
`locales/{de,en}/LC_MESSAGES/base.po` (translations), built with
[Babel](https://babel.pocoo.org/) — a pure-Python replacement for
`xgettext`/`msgfmt`/`msginit` (`pip install babel`, or `pip install
'pbd-toolkit[dev]'`).

```bash
python scripts/update_catalog.py extract   # re-extract locales/base.pot from source
python scripts/update_catalog.py update    # merge new/changed strings into de/en .po
# ... translate any new msgid in locales/de/LC_MESSAGES/base.po by hand,
#     or add it to scripts/_i18n_de_translations.py and run fill-de ...
python scripts/update_catalog.py fill-de   # apply scripts/_i18n_de_translations.py
python scripts/update_catalog.py fill-en   # set English msgstr = msgid
python scripts/update_catalog.py compile   # compile .po -> .mo (what gettext actually loads)
python scripts/update_catalog.py check     # CI gate: fails on drift or untranslated entries
```

`extract` scans the files listed in `babel.cfg` (`core/cli.py` and
`core/scan_reporting.py` today) for calls to `_(...)` or `translate_func(...)`.
The `i18n` CI job runs `check` on every push and fails if source strings
were added without re-extracting, or if a `.po` entry is fuzzy/untranslated.

> **Run the catalog tooling under Python 3.12** (the interpreter CI uses).
> Babel's Python-source extractor finds `_()`/`translate_func()` calls
> embedded inside f-strings (e.g. `f"{context._('Started:')}  {value}"`)
> differently depending on the Python version, because f-string tokenization
> changed in 3.12 (PEP 701). Running `extract`/`check` under an older
> interpreter can under-extract and desync `locales/base.pot` from what CI
> expects.

## Known follow-up (out of scope for this pass)

- Typer `help=` option text (~100 options on `scan`, dozens more across the
  other subcommands) is not yet translated — only echoed messages and errors
  are. Wrapping `help=` requires checking how Typer/Click evaluate decorator
  defaults (import-time vs. call-time), since the active locale can only be
  known once `LANGUAGE` is read.
- `core/doctor.py`'s issue messages (`doctor` command) are not yet routed
  through the translator.
- `core/cli_setup.py`'s dead argparse path (`__setup_args`) still contains
  the old gettext-wrapped strings; it is intentionally excluded from
  extraction here and is slated for deletion in the CLI decomposition work
  (see the architecture roadmap), at which point `locales/base.pot`'s
  `X-Poedit-SearchPath` cruft referencing the old `main.py` can also go.
